Pricing Simulator App Tech Spec
Version: MVP v1
Target deployment: Render
Primary goal: Learn how customer-level pricing behavior and later pricing experiments affect orders, revenue, and profit over a 90-day simulation.

==================================================
1. PRODUCT GOAL
==================================================

Build a simple web application with a frontend and Python backend that simulates a delivery-platform-style business.

The application must:
- simulate individual customers over 90 virtual days
- advance simulation time in 1-day steps
- run an initial baseline period with no experiments
- later introduce pricing experiments
- model customer purchase decisions statistically
- store and display results for analysis

This is a learning and experimentation tool, not a production marketplace system.

==================================================
2. CORE PRODUCT REQUIREMENTS
==================================================

The app must support two phases:

A. Baseline phase
- run under default pricing
- no experiments active
- customer behavior evolves naturally
- customer history is built during this period

B. Experiment phase
- begins on a configurable simulation day
- pricing experiments are introduced
- customers are assigned to control or variant groups
- outcomes are measured and compared

Recommended default timeline:
- Days 1-30: baseline
- Days 31-90: experiment

==================================================
3. CORE DOMAIN MODEL
==================================================

The simulator is a customer-level stochastic simulation.

Unit of simulation:
- one customer

Time unit:
- one day

Simulation horizon:
- 90 days

Main idea:
- instantiate many customer objects
- each customer has fixed traits and dynamic state
- on each day, each customer may or may not purchase
- purchase probability depends on price and customer state
- after experiments start, pricing may differ by treatment group

==================================================
4. CUSTOMER MODEL
==================================================

Each customer must be represented as a class instance.

4.1 Static customer attributes
These are assigned when the customer is created.

Required fields:
- customer_id: unique identifier
- budget: maximum spending level the customer can tolerate
- buy_propensity: baseline tendency to buy
- price_threshold: price level below which customer is more likely to buy
- repeat_boost: multiplier or additive effect for repeat-purchase tendency
- segment: optional label such as new, loyal, price_sensitive
- basket_mean: expected order value if purchase occurs
- location_zone: optional area or market zone

Minimal required pricing traits:
- budget
- buy_propensity
- price_threshold

4.2 Dynamic customer state
These evolve during the simulation.

Required fields:
- has_purchased_before: boolean
- purchase_count: number of prior purchases
- last_purchase_day: most recent purchase day or null
- days_since_last_purchase: derived recency metric
- assigned_treatment: control or variant after experiment starts

4.3 Customer behavior requirement
New customers must be harder to convert.
Returning customers must be more likely to buy again.

This means the purchase model must include:
- a lower probability for customers with no previous purchase
- a higher probability for customers with prior purchases
- an update to customer state after each successful purchase

==================================================
5. MINIMAL PURCHASE LOGIC
==================================================

The purchase decision must be probabilistic, not deterministic.

For an offered total price:

Case 1:
- if price > budget
- purchase probability = 0 or near 0

Case 2:
- if price <= budget but price > price_threshold
- purchase probability is reduced

Case 3:
- if price <= price_threshold
- purchase probability is higher

Repeat-purchase effect:
- if customer has purchased before, probability increases
- this increase can depend on purchase_count and/or recency
- keep the first version simple and capped

Recommended conceptual form:

purchase_probability
= base propensity
  x price effect
  x repeat effect
  x optional calendar effect

where:
- price effect decreases as price rises
- repeat effect is lower for new customers and higher for returning customers
- probability must be clipped to [0, 1]

==================================================
6. PRICING MODEL
==================================================

The pricing model should start minimal.

6.1 Default pricing policy
Used during baseline phase.

Required fields:
- item_price or order_price
- delivery_fee
- service_fee
- optional discount amount
- optional free-delivery threshold

6.2 Experiment pricing policy
Used during experiment phase.

Examples:
- lower delivery fee
- service fee change
- fixed discount
- free delivery above threshold

At MVP stage, one pricing lever is enough.
Recommended first lever:
- delivery fee

==================================================
7. EXPERIMENT MODEL
==================================================

Experiments must not start on day 1.

7.1 Baseline requirement
- run baseline with no active experiment
- preserve customer history from baseline into experiment phase

7.2 Experiment start
At a configured day:
- activate experiment
- assign customers to treatment groups

7.3 Treatment groups
Minimum:
- control
- one variant

Example:
- control: delivery fee = 2.99
- variant: delivery fee = 1.99

7.4 Assignment level
For MVP:
- assign at customer level
- keep customer in same treatment for remainder of experiment

==================================================
8. SIMULATION ENGINE
==================================================

The simulation engine is the core backend component.

8.1 Main responsibilities
- create customer population
- advance the simulation day by day
- apply baseline or experiment pricing depending on current day
- compute customer purchase probabilities
- sample purchases statistically
- generate order values when purchases occur
- update customer state
- record daily and cumulative metrics

8.2 Daily simulation loop
For each day:
1. increment simulation day
2. determine active pricing policy
3. for each customer:
   - compute offered price
   - compute purchase probability
   - sample buy/no-buy outcome
   - if buy:
     - sample or assign order value
     - compute revenue and margin
     - update customer state
4. aggregate day-level metrics
5. persist outputs

==================================================
9. STATISTICAL REQUIREMENTS
==================================================

The app must be statistical in nature.

Required properties:
- customer traits can be sampled from distributions
- purchases are sampled probabilistically
- different runs can produce different outcomes
- experiment results should be estimated from simulated data

Recommended first distributions:
- budget: normal, log-normal, or truncated normal
- buy_propensity: beta or bounded random draw
- price_threshold: derived from budget or sampled separately
- basket_mean: log-normal or gamma-like positive distribution

Recommended first estimation outputs:
- control vs variant conversion
- lift in orders
- lift in revenue
- lift in contribution margin

Optional later:
- confidence intervals
- hypothesis tests
- repeated simulation runs across seeds

==================================================
10. ORDER, REVENUE, AND PROFIT MODEL
==================================================

The simulator must record basic business outcomes.

Per purchase, compute:
- order_count = 1
- gross revenue
- discount amount
- fees charged
- net revenue
- variable cost
- contribution margin

Minimal MVP outputs:
- orders
- gross revenue
- discount spend
- net revenue
- contribution margin

Required metrics:
- total customers

- orders
- conversion rate
- average order value
- gross revenue
- discount spend
- net revenue
- contribution margin

Required breakdowns:

==================================================
15. DEPLOYMENT REQUIREMENTS
==================================================

Deploy on Render.

Recommended deployment shape:
- one web service for frontend/backend app
or
- one backend web service + static frontend

Simplest MVP:
- backend serves API
- frontend served from same app if convenient

Required infrastructure:
- Render web service

- environment variables for database connection and settings

Backend must support:
- production start command
- database migrations
- persistent PostgreSQL connection

==================================================
16. MVP SCOPE
==================================================

The MVP must stay small.

Include:
- one market
- one merchant/business type
- one customer population
- one baseline period
- one experiment starting later
- one pricing lever
- one control and one variant
- one 90-day run
- basic dashboards

Do not include in MVP:
- real maps
- real merchant catalog
- routing/logistics simulation
- multiple overlapping experiments
- causal inference complexity beyond simple treatment comparison
- production authentication unless needed

==================================================
17. NON-NEGOTIABLE LOGIC RULES
==================================================

The implementation must satisfy all of the following:

1. The simulation runs in daily steps.
2. The total virtual time is 90 days by default.
3. The app starts with a baseline period with no experiment.
4. Pricing experiments are introduced later, not at the start.
5. Each customer is a class instance with fixed pricing-related traits.
6. Each customer must include at least:
   - budget
   - buy_propensity
   - price_threshold
7. Customer state must evolve over time.
8. New customers must be harder to convert than returning customers.
9. Prior purchases must increase future purchase probability.
10. Purchase outcomes must be stochastic.
11. Results must include orders, revenue, and contribution margin.
12. The app must be deployable on Render using PostgreSQL.

==================================================
18. SUGGESTED PYTHON CUSTOMER INTERFACE
==================================================

The implementation should conceptually support a customer class like this:

Customer
- attributes:
  - customer_id
  - budget
  - buy_propensity
  - price_threshold
  - repeat_boost
  - basket_mean
  - has_purchased_before
  - purchase_count
  - last_purchase_day
  - assigned_treatment

- methods:
  - compute_purchase_probability(offered_price, current_day, context)
  - decide_purchase(random_state, offered_price, current_day, context)
  - register_purchase(current_day, order_value)

Exact code can vary, but this behavioral interface must exist.

==================================================
19. FIRST IMPLEMENTATION ORDER
==================================================

Build in this order:

1. Define database schema
2. Implement Customer class
3. Implement pricing policy objects
4. Implement daily simulation loop
5. Implement baseline-to-experiment transition
6. Implement metrics aggregation
7. Expose backend API
8. Build frontend scenario form
9. Build results dashboard
10. Deploy to Render

==================================================
20. ACCEPTANCE CRITERIA
==================================================

The MVP is acceptable if:

- a user can configure a run from the UI
- the backend generates a customer population
- the simulation advances 90 days in 1-day steps
- the first part runs with no experiment
- the later part runs with a pricing experiment
- customers have budget, buy_propensity, and price_threshold
- returning customers become easier to convert
- outputs are saved in PostgreSQL
- results are viewable in the frontend
- the app deploys successfully on Render

End of spec.
